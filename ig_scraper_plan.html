1Vue d'ensemble — ce que fait le système

Un worker automatisé se connecte avec TON compte Instagram existant, visite les comptes que tu as ajoutés dans un panneau admin (L'Étoile, Bikini, Bodega, etc.), prend un screenshot de chaque nouveau post, puis envoie ce screenshot à l'IA (Claude Haiku) qui extrait automatiquement titre/date/lieu/catégorie. L'event est ensuite inséré dans user_events avec la bonne ville.

Pourquoi c'est safe : pas de création de comptes, pas de mass-follow, pas de publication automatique, pas de spam. Juste un compte qui scrolle et prend des captures — un pattern que des milliers d'humains ont sans se faire bannir.

Détail du cron worker (3× par jour)

1
Login sur @ton_compte (session persistée, on ne se re-log pas à chaque run)
2
Lit la table ig_followed_accounts (active=true)
3
Pour chaque compte : visite le profil, récupère les 10 derniers posts
4
Ignore les posts déjà en DB (dédup par ig_post_id)
5
Pour chaque nouveau post : screenshot → Storage → scan IA → event créé

3Structure DB (2 tables à créer)

Table 1 — Les comptes IG à surveiller

CREATE TABLE ig_followed_accounts (
  id               BIGSERIAL PRIMARY KEY,
  handle           TEXT NOT NULL UNIQUE,     -- 'etoile_toulouse'
  ville            TEXT NOT NULL,            -- 'Toulouse'
  active           BOOLEAN DEFAULT true,
  last_checked_at  TIMESTAMPTZ,
  notes            TEXT,
  created_at       TIMESTAMPTZ DEFAULT now()
);

Table 2 — Historique des posts scrapés

CREATE TABLE ig_scraped_posts (
  id              BIGSERIAL PRIMARY KEY,
  ig_handle       TEXT NOT NULL,
  ig_post_id      TEXT NOT NULL,           -- shortcode IG pour dédup
  ville           TEXT NOT NULL,
  posted_at       TIMESTAMPTZ,
  scraped_at      TIMESTAMPTZ DEFAULT now(),
  screenshot_url  TEXT,
  scan_event_id   TEXT,                    -- FK vers user_events
  scan_status     TEXT DEFAULT 'pending',  -- pending/scanned/skipped/failed
  scan_error      TEXT,
  UNIQUE(ig_handle, ig_post_id)
);
